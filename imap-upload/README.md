# imap-upload — заливка писем .eml на любой IMAP-сервер

Заливает `.eml`-файлы, выгруженные скриптом [`yandex-dump`](../yandex-dump),
на новый почтовый сервер по IMAP (`APPEND`), с сохранением папок, флага
прочитанности и исходной даты писем.

Использует только стандартную библиотеку Python — ничего ставить не
нужно.

## Для каких почтовых провайдеров подходит

Для **любого сервера с доступом по IMAP** — скрипт использует только
стандартные IMAP-команды (`APPEND`, `LIST`, `NAMESPACE`) и не привязан
к конкретному хостингу. Типовые адреса серверов:

| Провайдер | IMAP-хост | Порт |
|---|---|---|
| Beget | `imap.beget.com` | 993 |
| Timeweb | `imap.timeweb.ru` | 993 |
| reg.ru (почта хостинга) | `mail.hosting.reg.ru` | 993 |
| Mail.ru / VK WorkMail (домены) | `imap.mail.ru` | 993 |
| Fastmail | `imap.fastmail.com` | 993 |
| Zoho Mail | `imap.zoho.com` / `imap.zoho.eu` | 993 |
| Gmail / Google Workspace | `imap.gmail.com` | 993 |
| Свой сервер (Dovecot, mailcow, Mailu, Mail-in-a-Box) | ваш хост | 993 |

Точный адрес IMAP-сервера смотрите в панели управления или справке
вашего провайдера — таблица выше может устареть. У некоторых
провайдеров (Gmail, Mail.ru, Fastmail, Zoho) для IMAP нужен отдельный
«пароль приложения», а не основной пароль аккаунта — он включается в
настройках безопасности аккаунта.

## Требования

- Python 3.8+ (внешних зависимостей нет — только стандартная
  библиотека)
- Каталог с результатом `yandex-dump` (там должен лежать
  `manifest.jsonl`)
- Доступ по IMAP к целевому ящику (хост, логин, пароль)

## Быстрый старт

```bash
cd imap-upload
./upload.sh /путь/к/dump box@ваш-домен.ру
```

`upload.sh` спросит адрес IMAP-сервера (или возьмёт из переменной
окружения `IMAP_HOST`), затем сначала всегда делает **dry-run** —
покажет, какие папки куда поедут и сколько писем в каждой, без единой
реальной записи на сервер — и только после вашего подтверждения
запускает заливку. Пароль запрашивается отдельно (`getpass`, не
отображается на экране и не пишется в историю шелла) — либо можно
передать его через переменную окружения `IMAP_PASS`.

Напрямую через Python:

```bash
python3 imap_upload.py --dump /путь/к/dump --user box@ваш-домен.ру \
  --host imap.example.com --dry-run
python3 imap_upload.py --dump /путь/к/dump --user box@ваш-домен.ру \
  --host imap.example.com
```

## Маппинг папок

Стандартные папки Яндекса переименовываются в стандартные IMAP-папки:

| Яндекс | IMAP |
|---|---|
| Входящие / Inbox | `INBOX` |
| Отправленные / Sent | `Sent` |
| Черновики / Drafts | `Drafts` |
| Спам / Spam | `Spam` |
| Удалённые / Trash | `Trash` |
| Архив / Archive | `Archive` |

Остальные (пользовательские) папки создаются на сервере с тем же
именем, включая вложенность и кириллицу (кодируется в IMAP UTF-7
автоматически).

**Namespace-префикс.** Многие серверы на Dovecot (в том числе Beget)
держат все папки внутри namespace с префиксом `INBOX.`: папка
отправленных на самом деле называется `INBOX.Sent`, а не `Sent`.
Попытка писать в `Sent` напрямую даёт ошибку вида:

```
Client tried to access nonexistent namespace.
(Mailbox name should probably be prefixed with: INBOX.)
```

Скрипт определяет префикс автоматически (IMAP-команда `NAMESPACE`)
при подключении и сам подставляет его ко всем папкам, кроме `INBOX` —
руками ничего указывать не нужно. Определённый префикс печатается в
строке «Подключено к ...» в начале заливки.

Переопределить маппинг конкретной папки:

```bash
python3 imap_upload.py --dump /путь/к/dump --user box@example.com \
  --host imap.example.com \
  --folder-map 'Рассылки=Newsletters' --folder-map 'Заказы=INBOX.Заказы'
```

## Если в ящике на новом сервере уже есть письма

Например, если MX-записи домена уже переключены на новый сервер и туда
успела прийти часть свежей почты, пересекающейся с архивом — обычный
`APPEND` создаст дубликаты, IMAP это никак не проверяет сам.

Флаг `--dedupe` решает это: перед заливкой в каждую папку скрипт
разово вычитывает у сервера заголовки `Message-ID` уже лежащих там
писем (без скачивания тела, батчами — быстро даже на тысячах писем) и
пропускает совпадения:

```bash
python3 imap_upload.py --dump /путь/к/dump --user box@example.com \
  --host imap.example.com --dedupe
```

Если ящик на новом сервере создан только что и точно пуст — флаг не
нужен, он просто немного замедлит старт.

## Резюмируемость

Список уже успешно залитых писем хранится в
`<dump>/uploaded_<логин>.txt`. Прервали — запустите ту же команду
снова, дольёт только недостающее.

## Все параметры

| Флаг | Что делает |
|---|---|
| `--dump DIR` | каталог с выгрузкой (обязателен, должен содержать `manifest.jsonl`) |
| `--user LOGIN` | логин ящика на новом сервере (обязателен) |
| `--host` | IMAP-хост назначения (обязателен, см. таблицу провайдеров выше) |
| `--port` | IMAP-порт (по умолчанию `993`, SSL) |
| `--dry-run` | только показать план по папкам, ничего не заливать |
| `--dedupe` | пропускать письма, чей `Message-ID` уже есть в целевой папке |
| `--folder-map "Откуда=Куда"` | переопределить назначение конкретной папки (можно несколько раз) |
| `--delay 0.05` | пауза между заливками, сек |

Пароль: переменная окружения `IMAP_PASS`, либо интерактивный запрос
при отсутствии переменной.

## Планы на будущее

Скрипт сознательно устроен максимально сервер-агностично (обычный
IMAP APPEND) — расширение под провайдеров, у которых заливка устроена
иначе (не по IMAP, а через собственный API), потребует отдельного
скрипта в духе `<provider>-upload/`, но переиспользует всё, что не
зависит от протокола (чтение `manifest.jsonl`, маппинг папок,
дедупликация по `Message-ID`).
