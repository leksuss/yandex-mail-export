# yandex-dump

Выгружает все письма из Яндекс.Почты в файлы `.eml`, **не используя
IMAP/SMTP** — через тот же внутренний веб-API, которым пользуется сам
`mail.yandex.ru`, когда вы открываете почту в браузере.

Зачем это нужно: с 29 июня 2026 Яндекс закрыл IMAP/SMTP-доступ для
бесплатных ящиков на собственных доменах. Веб-интерфейс при этом
продолжает работать бесплатно — этот скрипт использует его вместо
IMAP.

## Как это работает и почему это безопасно

Скрипт **не логинится в Яндекс и не знает вашего пароля**. Вместо этого
вы вручную копируете cookies уже открытой в браузере сессии — так же,
как это делают расширения для экспорта cookies. Это осознанное решение:

- Скрипт физически не может увидеть или сохранить ваш пароль — его
  просто нет ни в одном месте выполнения.
- Единственный домен, с которым скрипт вообще взаимодействует —
  `mail.yandex.ru` (это буквально одна константа `BASE` в начале файла
  `yandex_dump.py`, зайдите и проверьте сами).
- Никакой телеметрии, аналитики, сторонних запросов в коде нет.

Тем не менее относитесь к файлу с cookies как к паролю: по нему можно
зайти в ваш ящик без второго фактора. Не публикуйте его, не
коммитьте в git, удалите после использования.

## Требования

- Python 3.8+ (проверить: `python3 --version`)
- Модуль `venv` (входит в стандартную поставку Python; в Debian/Ubuntu
  может потребоваться `apt install python3-venv`)
- Пакет `requests` — единственная внешняя зависимость

**Если запускаете через `dump.sh` — ставить ничего не нужно**: при
первом запуске он сам создаст виртуальное окружение `.venv` рядом со
скриптом и поставит туда `requests`. Это работает и на системах, где
pip запрещает установку вне окружения (`PIP_REQUIRE_VIRTUALENV`,
externally-managed-environment / PEP 668).

Если запускаете `yandex_dump.py` напрямую, создайте окружение вручную:

```bash
cd yandex-dump
python3 -m venv .venv            # создать виртуальное окружение
source .venv/bin/activate        # активировать (Windows: .venv\Scripts\activate)
pip install requests             # поставить зависимости
```

Дальше в этом же терминале запускайте `python yandex_dump.py ...`.
Окружение активно только в текущем сеансе терминала: в новом сеансе
снова выполните `source .venv/bin/activate` (выйти из окружения —
`deactivate`). Каталог `.venv` уже добавлен в `.gitignore` — в git он
не попадёт.

## Быстрый старт

```bash
cd yandex-dump
./dump.sh
```

Скрипт `dump.sh` при первом запуске без `cookies.txt` сам покажет
пошаговую инструкцию, что делать. Коротко:

1. Откройте https://mail.yandex.ru в браузере, залогиньтесь.
2. DevTools (F12) → вкладка **Network**.
3. Кликните на любое письмо в списке.
4. В Network кликните на любой запрос к `mail.yandex.ru` → вкладка
   **Headers** → в Request Headers найдите заголовок `Cookie` →
   скопируйте его значение целиком.
5. Сохраните в файл `cookies.txt` рядом со скриптом:
   ```bash
   printf '%s' 'Session_id=...; yandexuid=...; ...' > cookies.txt
   ```
   Формат Netscape `cookies.txt` (как экспортируют браузерные
   расширения) тоже поддерживается — можно использовать его как есть.

`dump.sh` дальше сам: спросит адрес выгружаемого ящика (чтобы назвать
каталог выгрузки понятно, например `dump_box@example.com`, а не
безликим `dump`; можно и не отвечать — тогда будет просто `dump`, или
явно задать своё имя каталога вторым аргументом:
`./dump.sh cookies.txt мой_каталог`), проверит папки
(`--list-folders`), скачает одно письмо для проверки (`--probe`) и
только после вашего подтверждения запустит полную выгрузку.

Можно и без обёртки, напрямую через Python:

```bash
python3 yandex_dump.py --cookies cookies.txt --out dump --list-folders
python3 yandex_dump.py --cookies cookies.txt --out dump --probe
python3 yandex_dump.py --cookies cookies.txt --out dump
```

## Результат

```
dump/
  Inbox/<mid>.eml
  Sent/<mid>.eml
  <Ваша папка>/<mid>.eml
  manifest.jsonl        # список писем: mid, папка, прочитано ли, дата
```

`manifest.jsonl` нужен второму скрипту (`beget-upload`) для заливки на
новый сервер — он сохраняет папки, флаг прочитанности и исходную дату
письма.

Выгрузка **резюмируемая**: если прервать (Ctrl+C) или скрипт упадёт по
сети, просто запустите ту же команду ещё раз — уже скачанные письма
(по `manifest.jsonl`) не скачиваются повторно.

## Если у вас скрипт не работает "из коробки"

Внутренний веб-API официально не документирован и может отличаться в
зависимости от версии клиента, региона, типа аккаунта. Скрипт написан
защитно и подстраивается сам в большинстве случаев, но если что-то не
работает:

### Папки не находятся / письма не листятся

У Яндекса внутренняя модель API называется в основном `liza1`, но это
может отличаться. Проверка: DevTools → Network → отфильтруйте по
`models` → кликните на любой POST-запрос → посмотрите его URL, он
выглядит как `/web-api/models/<ИМЯ>?_m=...`. Если `<ИМЯ>` не `liza1` —
передайте его скрипту:

```bash
python3 yandex_dump.py --cookies cookies.txt --out dump --api-model ВАШЕ_ИМЯ
```

### `--probe` не может скачать письмо ("ни один шаблон URL не сработал")

Откройте любое письмо в вебе → кнопка «Свойства письма» → «Скачать
оригинал» → это скачает `.eml` через браузер. В DevTools (вкладка
Network) найдите этот запрос и скопируйте его полный URL. Передайте
его скрипту как шаблон, заменив число письма на `{mid}`, uid — на
`{uid}`:

```bash
python3 yandex_dump.py --cookies cookies.txt --out dump --probe \
  --source-template 'https://mail.yandex.ru/ваш/URL/{uid}/{mid}/что-то.eml'
```

### Что-то ещё пошло не так

Запустите с `--debug` — сырые JSON-ответы API лягут в `dump/_debug/`.
В репозитории есть вспомогательный скрипт `inspect_debug.py` (в корне
репозитория), который печатает **структуру** такого JSON (пути ключей,
длины списков, счётчики) без содержимого писем — им удобно
диагностировать проблему, не публикуя переписку:

```bash
python3 ../inspect_debug.py dump/_debug/folder_1_0.json
```

## Все параметры

| Флаг | Что делает |
|---|---|
| `--cookies FILE` | файл с cookies (обязателен) |
| `--out DIR` | каталог для выгрузки (по умолчанию `dump`) |
| `--list-folders` | только показать список папок и выйти |
| `--probe` | скачать одно письмо для проверки и выйти |
| `--folders "Папка1" "Папка2"` | выгружать только перечисленные папки |
| `--delay 0.4` | пауза между запросами скачивания, сек |
| `--page 100` | сколько писем запрашивать за раз при листинге |
| `--api-model liza1` | имя внутренней модели API (см. выше) |
| `--source-template URL` | свой шаблон URL оригинала письма (см. выше) |
| `--ckey ...` / `--uid ...` | передать CSRF-токен/uid вручную, а не получать автоматически со страницы |
| `--debug` | сохранять сырые JSON-ответы API в `<out>/_debug/` |

## Известные особенности API, которые скрипт уже обходит

Ничего из этого не требует ваших действий — просто для понимания, что
происходит под капотом:

- **CSRF-токен (`ckey`) периодически "протухает"** посреди сессии и
  содержит символы, которые в HTML-странице приходят в
  JSON-экранированном виде (`\u002F` вместо `/`) — если их не
  раскодировать, все запросы с таким токеном будут молча отклоняться.
  Скрипт декодирует токен и переполучает его заново при первой ошибке.
- **Листинг папки отдаёт максимум ~30 писем за раз**, независимо от
  того, сколько запрошено — скрипт учитывает это и подбирает шаг
  пагинации по факту полученного, а не по запрошенному числу.
- На случай, если в вашем аккаунте листинг всё же тредный (шапки
  переписок вместо отдельных писем), скрипт умеет доразворачивать
  такие треды в отдельные письма — эта логика включается автоматически,
  если число писем в листинге меньше счётчика папки.
